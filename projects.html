Audio Web Visualizer

Summer 2018

This project was built in collaboration with Ryan Menghani and Gauraang Dhamankar at UT. We decided to create a web music visualizer, using MDN's Web Audio API to grab MP3 data and the P5 API to create easy graphics.

In particular, this version of the visualizer uses a particle system in which stationary dots are manipulated by their peers. This allowed the visualizer to simulate creative visuals like water waves or fizzy bubbles.

One of the features in this application is the use of a rudimentary beat detection; a biquad filter was used on the frequency data analysed from the song and was averaged - when the amplitude of a low frequency was large enough, a "beat" was detected and the visualizer draws a pattern on the canvas (imagine a box cordoned into sections, drawing a pattern means pouring water into some sections and then opening the sections, releasing the water).

Another feature is a separate mode, based off of cellular automata. It's very similar to Conway's Game of Life, in which a particle "lives" based on whether 3 or 4 neighboring cells are alive. This gives the visualization a unique shimmering effect where particles warp in and out of existence through their size.

Checkout the visualizer here!

On one of the first iterations of this project, I intended to make a pseudo- physics particle system in order to emulate waves, but that turned out south after considering the ramifications (interactions with > 2 particles, size vs field of influence, etc). But I did end up with something cool at least!

Generative Drawings

Summer 2018

This was a quick, couple day project I did at the end of July. I've recently been intrigued by generative art after the Web Audio Visualizer project, and wanted make my own little programs that created designs based off of "natural" systems.

In particular, I wanted to model gravitational attraction (being in EM physics class for summer credit right now), and so built classes of objects that moved based on vector acceleration (This was a problem particularly for the particle system I tried to implement for the visualizer earlier).

Here are five that I've made right now; you can right click on the titles to access the browser window/codepen and inspect them to see the source code. You can also right click and save the screen as an image to use as a background or screensaver.


Orbits

This one in particular is a simulation of particles under gravity (by each other, or by a stationary "black hole"). Thirty "planets" are spawned over time, and they are attracted to each other to form semi-regular orbits. If you uncomment in the Gravity_Bodies.js code lines 45-48 (with the strokes and lines) in the inspector, you can see scaled vectors of the velocity and acceleration.


Traces

Traces is a modified version of Pointless, where the background has a a reduced opacity, allowing trails of the particles to fade away slowly as they fly across the window. In addition, the acceleration vectors are reduced, minimizing curving of the objects.


Pointless

Pointless (ha ha) is the first iteration I made, in which very fast particles move around a static canvas. They have a larger acceleration than Traces, allowing them to curve back on themselves if they bounce off the walls.

FlowField

This was a fun demo to make, using Perlin noise to direct particles across the canvas to flow in various directions, like water currents. I followed around with the base code with Dan Shiffman's Coding Train tutorial series, but split up around the middle to build my own implementation. It was very interesting to see how similar our end simulations ended up, although we had slightly different ideas on how to implement movement and boundary checking.

For this demo, there was really a lot of ways to create different representations of the system, from purely displaying the flowfield as like a heatmap to showing particle flows and their traces like a kind of organic tunnel system. Make sure to check it out! I think it would be interesting to make a similar simulation, but instead of regions of differing vector directions, have regions of different "height" or "pressure", forcing particles to flow in one general directions, such as "down a hill and into a basin."

Noisy

This project was done for the purpose of learning how to access and modify the pixels array that P5.js uses to represent its canvas display. I decided to take what I learned about Perlin Noise from the FlowField demo and run images through a filter.

I used the following algorithm:
  1. Create a threshold by first passing all pixels through a loop and grabbing the average of their RGB values.
  2. Use that first threshold and run through the pixel array again, but for each pixel's R,G,B value, check if they exceed the threshold. If they do, apply a modifier created by grabbing noise values at the start of each block of pixels (of arbitrarily chosen size).
  3. After passing through the image, update the image with the new RGB values, and continue iterating until paused.

Step two previously had the noise recalculated for every pixel, but that posed a problem of being computationally intensive (to the point of crashing Chrome). This was solved by reapplying the noise for every set of pixels, whose size strongly determines the resulting design.

For example, there is a comment in the codepen that suggests the value 19 in the line

[68] ...if( i/(d*width/(width/19)) == blockIdx ) {...}

can be adjusted to 25 or 100. If the value is 100, the resulting image is like the top left image in the first slide of the slideshow, with horizontal bars of "static" across similar regions. If the value is 19 or 25, the tendency is to have a more blob or fine grain texture in the static. In fact, this is true for all values indivisible cleanly by 4. Values such as 24, or 80, produce visually similar images to 100. It will take me a little bit to ponder why this is the case, but it possibly might have to do with the fact that the index, i, increases by 4 for every iteration since each pixel uses 4 indices for RGBA values.

Another unexpected result of this demo is that drawings, like web art, do not produce as fragmented pictures. This is likely because pictures hold a lot more contrast across the image where web art has more even distributions of flat colors.

Application wise, this would be really cool as a noise filter for images. Looking through GIMP's noise filters, there isn't anything like this that applies noise in correlation with the current picture color. However, it should be acknowledged that there is a challenge in picking good values for number of iterations, since at some point the image is blurred out beyond recognition going through so many passes.

Markov Chain Image Generation

January 2019

The second half of my winter break was spent working on this project, a picture generator based on Markov Chains. My initial idea for this project came from when I was reading this cool article where the author uses Markov chains to build sentences and extrapolate that process to art.

The concept behind Markov Chain data generation is pretty simple; a datapoint (in my case, a point in the color space) is associated to other datapoints based on the statistical possibility of their occurrence in sequential order.

For example, if the word "hello" is followed by the word "world" in a sentence, then when the word "hello" is typed again, there is a non-insignificant chance that "world" will be the next word. In the case that the dataset that has every "hello" instance followed by a "world" instance, that chance is 100%.

You can quite possibly imagine the implications of such a concept. In fact, Markov chains are implemented in a variety applications such as sentence completion, financial modeling, and even in PageRank, Google's famous algorithm that has a dark hand in the billions of search queries made by Googlers every day.

In the first version of my program, I make use of a discrete-time Markov chain by building a two dimensional matrix; the x axis represents the current state and the y axis represents the probability distribution of the next state.

I fill this matrix by reading pixel data from images in a dataset. When finished, the distribution in the matrix presents an interesting (but in hindsight, expected) result: along the y axis of the matrix there are indices that display outsized probabilities of the colors at that index occurring.

Looking back at the "hello world" example, this makes sense. Images of objects or backgrounds, like phrases, are not random. There is a sense of order, whether in the word order or whether the sky is blue and white (and not green), in which this memoryless process can capture.

However, when outputting the initial results of the algorithm designed, it became clear that while the program can adequate represent an image set's outstanding colors, it wasn't quite possible to replicate the basic structures that would allow the human brain to recognize as an identifiable object.

The solution lies back in the previously mentioned Magenta article; by relying on a larger order (like a 2nd order differential equation), low level structural randomness can be reduced to higher level randomness: for example, the petals of a sunflower may become recognizable but are scattered across the page. If the order is large enough, the probability matrix collapses into a single image, which may be a replica of a pattern in the input data. A good blend between the two is what's needed in order to create interesting pictures.

However, larger order pictures present a spatial and temporal constraint for the current implementation. With just a 2nd order markov chain, looking at the left and above pixel, the data structure needed to look through for the index is n^3. Consider a cube. The length is the 1st index for the left pixel, the width is the 2nd index for the above pixel, and the height contains the probabilities for the possibilities of the current pixel's color at the 1st and 2nd index.

Increasing the order (adding another parameter as another index) makes the size of the data structure increase from n^3 to n^4, which is simply unsustainable at high orders.

Current and Future Plans

As of recently, I'm still considering modularizing this program in order to create arbitrary n-order Markov Chains with images. My efforts in optimizing the program last January has resulted in very significant efficiency gains, but only work so far with quantized color buckets of x colors and with a 2nd order Markov Chain. Further progress will require a restructuring of the data structure to remove empty or near empty data points in the n-dimensional "box", meaning that each dimension doesn't have to be limited to strictly x quantized colors and can be less.

A possible idea is to use chained linked lists with some hashing function that relates the colors (in Markov::step()) to the new linked lists (previously they were associated with the index in the x color length dimension, which was very handy).
